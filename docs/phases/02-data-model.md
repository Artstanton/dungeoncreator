# Phase 2 — Data Model + Persistence

## What was built

- **Prisma schema** (`apps/api/prisma/schema.prisma`) — four models: `Campaign`, `Dungeon`, `Level`, `Room`. SQLite via Prisma 5.
- **Prisma singleton** (`apps/api/src/lib/prisma.ts`) — hot-reload-safe client (attaches to `globalThis` so `tsx watch` doesn't leak connections).
- **Seed script** (`apps/api/prisma/seed.ts`) — creates two campaigns and a fully-populated sample dungeon (2 levels, 3 rooms) without calling Grok.
- **Shared Zod schemas** (`packages/shared/src/index.ts`) — `Campaign`, `CampaignListItem`, `Dungeon`, `DungeonDetail`, plus `createCampaignInput` and `createDungeonInput` for validated POST bodies.
- **Campaign routes** (`apps/api/src/routes/campaigns.ts`):
  - `GET /api/campaigns` — list all campaigns with dungeon count.
  - `POST /api/campaigns` — create a campaign; 409 on name collision.
  - `GET /api/campaigns/:id` — single campaign with its dungeon list.
- **Dungeon routes** (`apps/api/src/routes/dungeons.ts`):
  - `GET /api/dungeons` — list all dungeons with campaign name and level count.
  - `GET /api/dungeons/:id` — full dungeon detail including campaign and levels summary; JSON columns (specificTreasures, specificEncounters) deserialized back to arrays.
  - `POST /api/dungeons` — create a dungeon; accepts either `campaignId` (existing) or `campaignName` (auto-creates/upserts the campaign); generates a random seed if omitted.

## Decisions made

- **JSON stored as `String`** — SQLite has no native JSON type. We explicitly `JSON.stringify()` on write and `JSON.parse()` on read, rather than relying on Prisma's `Json` type (which is string-backed anyway on SQLite and hides the round-trip). Keeping it explicit makes the schema self-documenting.
- **`direction` stored as `String`** — Prisma does support enums with SQLite but they're TEXT under the hood with no DB-level constraint. Storing as plain String and validating at the Zod layer is equivalent and simpler. The Zod `directionSchema` (`z.enum(['up','down','both'])`) enforces valid values at the route boundary.
- **Cascade deletes** — `Level` cascades from `Dungeon`; `Room` cascades from `Level`. Deleting a dungeon cleans up everything below it.
- **`postinstall` runs `prisma generate`** — so `pnpm install` always regenerates the TypeScript client. No separate manual step after pulling new schema changes.
- **No auth dropped** — Phase 1 docs mentioned `@fastify/session` for Phase 2. Removed per CONTEXT.md (no-auth decision, local-only app).

## How to run

One-time setup after pulling this branch:

```bash
# 1. Make sure root .env has DATABASE_URL set:
#      DATABASE_URL=file:../../data/dev.db
#    The Fastify runtime reads this from the repo root.
#    Prisma CLI reads from apps/api/prisma/.env (already committed with the same value).

# 2. Install deps and generate the Prisma client
pnpm install

# 3. Create the database + run the initial migration (shorthand script)
pnpm --filter @dungeon/api run db:migrate --name init

# 4. Seed with sample data
pnpm --filter @dungeon/api run db:seed

# 5. Start dev servers as usual
pnpm dev
```

Smoke-test the new routes while `pnpm dev` is running:

```bash
# List campaigns
curl http://localhost:4000/api/campaigns | jq

# Create a campaign
curl -s -X POST http://localhost:4000/api/campaigns \
  -H 'Content-Type: application/json' \
  -d '{"name":"My Campaign"}' | jq

# List dungeons (includes the seeded one)
curl http://localhost:4000/api/dungeons | jq

# Dungeon detail (replace <id> with the id from above)
curl http://localhost:4000/api/dungeons/<id> | jq
```

The Prisma Studio GUI is also available:

```bash
pnpm --filter @dungeon/api run db:studio
# Opens http://localhost:5555
```
